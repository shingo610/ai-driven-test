# Diagram Inbound Trend

**旅行会社の営業担当（自分）**が、インバウンド分野の未経験状態から、毎朝の短時間インプットで「覚える→引き出す」を回せるように、記事やメモを **単一HTML（Tailwind + Lucide）** の図解ダッシュボードに落とし込む。

## ヒアリングで固めた前提（SSOT）

| 項目 | 内容 |
|------|------|
| 利用シーン | 毎朝の通勤時間（10〜20分）にスマホで読む／必要に応じて昼に見返す |
| 読む人 | 自分だけ（インバウンドは初めてで、知識の地図がない） |
| 困りごと | 情報が散らばり「何が重要か」「どう会話に使うか」に変換できない |
| 成功の定義 | 最新トレンドを見て、**社内・お客様との会話で思い出せる“引き出し”**として使える（用語・スポット・一言ストーリーがセットで出る） |

## ワークフロー

### Phase 0: リサーチ（任意・サブエージェント）

トピックに地名・制度・統計など外部知識が絡む場合だけ、短時間で誤解を減らす目的で調べる。

```
Task({
  subagent_type: "explore",
  description: "インバウンドトレンド題材の調査",
  prompt: `
    以下について、旅行会社の営業が会話で使える粒度で箇条書きにしてください。
    【題材】{URL / 記事のコピペ / 自分のメモ}

    1. 何が「新しい」ポイントか（昨年までと何が違うか）
    2. どんな訪日客（国・年代・目的）が関係しやすいか
    3. いつ強いか（季節・曜日・時間帯・繁忙期）
    4. 注意点（過度な一般化、混雑、規制、炎上ポイント）
    5. 会話に変換する一言（30秒で言える）
  `
})
```

### Phase 1: ソース読み込み

1. ユーザーが渡した **記事URL／記事本文／SNS投稿の要約／自分の箇条書きメモ** を読む
2. 「何がトレンドか」を **1行で定義**（例: “◯◯が増えている／◯◯が人気”）
3. **利用文脈**（誰が・いつ・なぜ）を抽出（不明なら仮で置く）
4. **会話での使い道**（社内向け／顧客向け／仕入れ・提案向け）を1つ以上決める

### Phase 2: 用語翻訳表を作成

インバウンド特有の用語（OTA/DMO/ゴールデンルート等）と、記事固有の略語を洗い出し、[references/term-dictionary.md](references/term-dictionary.md) の形式で「やさしい言葉＋たとえ」を付ける。

### Phase 3: 「覚える3つ」を決める（引き出し化）

通勤時間で覚えやすいように、必ず **3つ** に圧縮する。

- **スポット／体験**: どこ・何をするか（固有名詞は1つまで）
- **理由**: なぜ今それが選ばれるか（背景・行動様式）
- **営業への変換**: どんな提案の引き出しになるか（誰に何を勧めるか）

### Phase 4: 会話テンプレ（30秒・2分）を作る

同じトレンドでも「短い会話」と「深掘り」の2段階を用意する。

- **30秒版**: 結論→理由→例（1つ）  
- **2分版**: 30秒版 + 注意点 + 代替案（混雑回避等）

### Phase 5: 情報を絞り込む（朝向け）

- 先頭に「**今日のインプット（10分）**」を置く（チェックリスト形式）
- 1画面目で **覚える3つ + 30秒会話** まで出す
- 詳細は後半に回し、数字や固有名詞の羅列を避ける

### Phase 6: キャラクター対話を設計（自分向け）

| キャラ | 役割 |
|--------|------|
| **いまの私** | 初学者の疑問を短く言う（何が大事？誰向け？） |
| **インバウンド先輩** | 重要点を3つに絞り、会話への落とし込みを提示 |

詳細 → [references/character-usage.md](references/character-usage.md)

### Phase 7: HTML生成

1. Tailwind CSS CDN + Lucide（**絵文字禁止**）
2. カラーパレットは [references/html-structure.md](references/html-structure.md) の「インバウンドトレンド用」変数を使用
3. 必須コンポーネント
   - **今日のインプット（10分）チェックリスト**
   - **覚える3つ**
   - **用語解説ボックス**
   - **会話テンプレ（30秒／2分）**
   - **いつ・誰に強いか（季節×客層のミニ表）**

### Phase 8: 批判的レビュー（任意・サブエージェント）

```
Task({
  subagent_type: "generalPurpose",
  description: "図解の批判的レビュー（旅行営業の自分視点）",
  readonly: true,
  prompt: `
    あなたは「旅行会社の営業（インバウンドは初学者）」です。
    以下のHTML図解を読んで、明日の通勤時間に使えるかを基準にレビューしてください。
    【HTMLファイルパスまたは本文】{path or excerpt}

    ## レビュー観点
    1. 10分で要点（覚える3つ）が頭に入るか
    2. 会話で使える“言い方”（30秒版）があるか
    3. 固有名詞・略語が置きっぱなしになっていないか
    4. 過度な一般化（国別断定等）のリスクが抑えられているか
    5. 追いかけるべき次のソース（追加で見るべき観点）が示されているか

    ## 出力形式
    ### 致命的な問題
    ### 改善すべき点
    ### 良かった点
    ### 総評（5段階＋一言）
  `
})
```

### Phase 9: 納品形態

- 既定: **単一 `index.html`** として保存（ユーザー指定パスがあれば従う）
- 日々蓄積する場合は、`diagrams/` 配下に `YYYY-MM-DD_トピック.html` で保存しても良い（ユーザー運用に合わせる）

## 品質チェックリスト

### 必須

- [ ] 「覚える3つ」がある（スポット／理由／営業への変換）
- [ ] 「30秒会話」がある（結論→理由→例）
- [ ] 「2分会話」がある（注意点・混雑回避なども含む）
- [ ] 初出の略語・専門用語に用語解説ボックスがある
- [ ] 「いつ・誰に強いか」が一目で分かる（ミニ表でOK）
- [ ] 10分で読める（情報過多になっていない）

### デザイン

- [ ] Lucide のみ（絵文字禁止）
- [ ] スマホでも読める（レスポンシブ）
- [ ] 1画面目で“今日の学び”が完結する（覚える3つ＋会話）
