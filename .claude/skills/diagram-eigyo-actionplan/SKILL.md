# Diagram Eigyo Action Plan

**旅行会社・営業職・課長**が作成したアクションプランや方針文書を、**課員全員（営業・営業支援）**が毎日使えるように、\n
「今日、何を優先するか」「誰にボールがあるか」「数字はどこを見るか」を即決できる **単一HTML（Tailwind + Lucide）** に変換する。

## ヒアリングで固めた前提（SSOT）

| 項目 | 内容 |
|------|------|
| 利用シーン | 毎日PCを立ち上げたら開く（朝会前・営業開始前の5分） |
| 困りごと | 文字だけだと理解しづらい → 別途パワポで補足していた |
| 成功の定義 | 課員が**日次ダッシュボード**を見て、優先行動を判断し、**自分の担当分野に向けてすぐ動ける** |

## ワークフロー

### Phase 0: リサーチ（任意・サブエージェント）

題材に**数値目標・新商品・法規・競合**など外部知識が絡む場合のみ、`explore` で不足を補う。

```
Task({
  subagent_type: "explore",
  description: "アクションプラン題材の調査",
  prompt: `
    以下について調査し、箇条書きで返してください。
    【題材】{トピック or 文書の要約}
    1. 業界・会社文脈で押さえるべき前提
    2. 課員が誤解しやすい点
    3. 図解で優先すべき「流れ」「役割」「時期」の候補
  `
})
```

### Phase 1: ソース読み込み

1. ユーザーが渡した**アクションプラン本文・箇条書き・スライドのテキスト**を読む
2. **ゴール（数値・期限・優先テーマ）**と**役割分担**を抽出
3. Phase 0 の結果があれば突合し、抜けを補う
4. 日次運用のために、各施策を「毎日／週次／月次／都度」に振り分ける（不明なら仮で良い）

### Phase 2: 用語翻訳表を作成

社内略語・KPI・旅行業界用語を洗い出し、[references/term-dictionary.md](references/term-dictionary.md) の形式で「やさしい言葉＋たとえ」を付ける。

### Phase 3: たとえ話を3つ以上

| 種類 | 例（旅行・営業） |
|------|------------------|
| 身近な例 | 添乗の行程表、カウンターでの受付フロー |
| 組織の例 | フロント／バックの役割分担 |
| 時間軸の例 | 四半期・繁忙期に合わせたマイルストーン |

### Phase 4: 情報を絞り込む

- 画面最上部に「今日やること（5分で決める）」を置く（チェックリスト形式）
- **「まず覚える3つ」**（最優先の方針・数字・判断基準）を決める
- **誰が何をするか**は必ず表で示す（担当別の見取り図）
- 詳細は後半へ。日次で読む前提なので「会議用の説明」は削り、行動に直結する情報だけ残す

### Phase 5: キャラクター対話を設計

| キャラ | 役割 |
|--------|------|
| **ヒナ**（新人営業） | 現場の素朴な疑問・不安を代弁 |
| **課長** | 方針の意図と優先順位を短く説明 |

**詳細** → [references/character-usage.md](references/character-usage.md)

### Phase 6: HTML生成

1. Tailwind CSS CDN + Lucide（**絵文字禁止**）
2. カラーパレットは [references/html-structure.md](references/html-structure.md) の「営業アクションプラン用」変数を使用
3. **用語解説ボックス**・**日次チェックリスト**・**担当マトリクス**を必ず含める
4. キャラ画像がない場合は **Lucide の `user` アイコン＋色**で代用（同ファイル内で統一）

**詳細** → [references/html-structure.md](references/html-structure.md)

### Phase 7: 批判的レビュー（サブエージェント）

```
Task({
  subagent_type: "generalPurpose",
  description: "図解の批判的レビュー（課員視点）",
  readonly: true,
  prompt: `
    あなたは「旅行会社の営業支援スタッフ」です。数字は扱うが、課長の意図までは毎回は追えない立場です。
    
    以下のHTML図解を読み、忖度なくレビューしてください。
    【HTMLファイルパスまたは本文】{path or excerpt}
    
    ## レビュー観点
    1. 自分の担当に落とし込めるか（何をすればいいかが分かるか）
    2. 用語・略語がそのまま残っていないか
    3. 朝に見て「今日の優先行動」を5分で決められるか
    4. 情報過多で日次に読み切れない部分はないか
    5. 図がなくても文章だけで足りる部分に図が足りていない箇所はないか
    
    ## 出力形式
    ### 致命的な問題
    ### 改善すべき点
    ### 良かった点
    ### 総評（5段階＋一言）
  `
})
```

### Phase 8: ブラッシュアップ

Phase 7 を反映。**致命的 → 改善**の順で修正。

### Phase 9: 納品形態

- 既定: **単一 `index.html`** としてプロジェクト内またはユーザー指定パスに保存
- 社内サーバ・静的ホスティングに載せる場合はユーザー指示に従う（サンプルの surge は必須としない）

## 品質チェックリスト

### 必須

- [ ] 初出の略語・専門用語に**用語解説ボックス**がある
- [ ] **たとえ話が3つ以上**ある
- [ ] **ヒナ × 課長**の対話が導入・中間・まとめに入っている
- [ ] **「まず覚える3つ」**がある
- [ ] **今日やること（チェックリスト）**が先頭にある
- [ ] **担当・役割**が表または図で分かる

### デザイン

- [ ] Lucide のみ（絵文字禁止）
- [ ] スマホでも読める（レスポンシブ）
- [ ] キャラは画像または Lucide アバターで統一
